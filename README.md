# Spec Template Normalizer (Phase 1)

Phase 1 analyzes the look and feel of an architect's `.docx` specification and publishes a validated bundle for Phase 2. It does not retag, normalize, or rewrite the source document.

The supported deliverable is one uniquely named `.phase1` directory. Phase 2 must receive and validate that whole directory; copying selected JSON or XML files is not a supported handoff.

## What Phase 1 does

For each run, the pipeline:

1. Copies the selected DOCX to a private, immutable working snapshot and verifies that the selected file did not change during the copy.
2. Safely unpacks the snapshot and reads paragraph text, existing styles, numbering, and package structure.
3. Sends a slim structural representation to the Anthropic classifier. Formatting XML is derived locally, not generated by the model.
4. Requires an explicit disposition for every paragraph in the classification universe: exactly one `styled` or `ignored` decision.
5. Builds `portable_styles.xml` separately from the source package. The source document and extracted snapshot are never retagged.
6. Captures the supported formatting environment, including styles, defaults, theme, settings, page layout, headers/footers, numbering, fonts, relationships, and package inventory.
7. Validates the registries, classification audit, exact source parts, artifact checksums, and bundle manifest.
8. Publishes the finished bundle with a same-filesystem directory rename so consumers do not see a partially written bundle.

Phase 2 is a separate application. It validates this bundle before applying the captured style system to another specification.

## Output bundle

The directory name includes the source hash and a run-unique token:

```text
<template>--<source-hash>--<run-token>.phase1/
```

It contains:

| Artifact | Purpose |
|---|---|
| `phase1_bundle_manifest.json` | Bundle format/version, producer and classifier identity, source identity, required artifact list, byte sizes, and SHA-256 checksums |
| `arch_style_registry.json` | CSI role-to-style mapping, exemplar indices, resolved formatting metadata, and numbering provenance |
| `arch_template_registry.json` | Bounded capture of the supported Word formatting environment |
| `classification_audit.json` | Exact styled/ignored/out-of-scope disposition for every slim paragraph, plus source and content fingerprints |
| `source_styles.xml` | Byte-exact `word/styles.xml` from the snapshotted source |
| `portable_styles.xml` | Generated stylesheet containing the source styles plus locally derived CSI styles for Phase 2 |
| `source_settings.xml` | Optional byte-exact `word/settings.xml`; present when that part exists in the source DOCX |

The bundle does not contain a normalized DOCX or a modified extracted package. Do not rename, edit, omit, or add files inside a published bundle: strict bundle validation rejects checksum mismatches, missing required artifacts, and unlisted files.

## Classification contract

The supported CSI roles are:

- `SectionID` -> `CSI_SectionID__ARCH`
- `SectionTitle` -> `CSI_SectionTitle__ARCH`
- `PART` -> `CSI_Part__ARCH`
- `ARTICLE` -> `CSI_Article__ARCH`
- `PARAGRAPH` -> `CSI_Paragraph__ARCH`
- `SUBPARAGRAPH` -> `CSI_Subparagraph__ARCH`
- `SUBSUBPARAGRAPH` -> `CSI_Subsubparagraph__ARCH`
- `END_OF_SECTION` -> `CSI_EndOfSection__ARCH`

Empty paragraphs, paragraphs that contain section properties, and table paragraphs are structurally out of scope. Every other candidate paragraph must appear exactly once in either `apply_pStyle` or `ignored_paragraphs`. Editor notes, copyright notices, and other non-CSI text are recorded as explicit ignored decisions rather than silently disappearing from coverage.

Coverage must be 100 percent over that styled-or-ignored universe. Targeted classifier follow-ups are bounded; unresolved indices fail the run. Phase 1 never fills a gap by copying a neighboring paragraph's style.

The model selects roles, dispositions, and exemplar paragraphs. It is not allowed to provide `pPr`, `rPr`, fonts, spacing, indentation, or other formatting. Portable styles are derived from source OOXML, preserve their source-style inheritance, and include the numbering information needed by Phase 2.

## Usage

### GUI

```bash
python gui.py
```

Select an architect template, enter the API key, choose an output root, and click **Run Phase 1**. The output selection is a parent directory; a new `.phase1` child directory is created for each successful run. The activity log reports the final path and coverage.

### Headless library API

`phase1_pipeline.run_phase1()` is the canonical orchestration entry point:

```python
from pathlib import Path
from phase1_pipeline import run_phase1

result = run_phase1(
    source_docx=Path("ArchitectTemplate.docx"),
    output_root=Path("phase1-output"),
    api_key="...",
)

print(result.bundle_dir)
print(result.manifest_path)
print(result.coverage)
```

The function returns only after the bundle has passed validation and been atomically published. On failure, no final bundle directory is published.

## Safety and bounded capture

Phase 1 treats a DOCX as untrusted package input.

- Package members must remain inside the extraction root; absolute paths, traversal, duplicate names, and symbolic links are rejected.
- Extraction is limited to 10,000 entries, 512 MiB total uncompressed data, 128 MiB per part, and a maximum compression ratio of 1,000.
- Relationship XML is validated. Internal targets must resolve to contained package parts. `TargetMode="External"` targets are recorded but never dereferenced.
- Header/footer media is limited to 16 MiB per asset and 64 MiB in aggregate.
- Malformed required XML, missing required parts, ambiguous classification, source changes during snapshot/staging, and contract mismatches fail closed.
- Classifier input is limited to an estimated 150,000 tokens for the current single-pass implementation.

`arch_template_registry.json` is a supported, normalized representation, not a byte-complete rendering virtual machine. Its `capture_policy` records normalization choices such as removing volatile rsid attributes and proofing markers. Use `source_styles.xml` and, when present, `source_settings.xml` when exact source bytes are required.

## What Phase 1 does not do

- It does not change the source DOCX.
- It does not insert `<w:pStyle>` tags into source paragraphs.
- It does not produce a replacement or normalized DOCX.
- It does not apply the template to a consultant specification; that is Phase 2.
- It does not fetch external relationship targets.
- It does not claim to preserve every possible Word feature outside the documented registry and exact companion parts.

## Validation and schemas

Formal contracts live in `schemas/`:

- `phase1_bundle_manifest.v1.schema.json`
- `classification_audit.v1.schema.json`
- `phase1_instructions.schema.json`
- `arch_style_registry.v2.schema.json`
- `arch_style_registry.v1.schema.json` (legacy registry version)
- `arch_template_registry.example.json` (reference shape)

Phase 2 should begin by validating `phase1_bundle_manifest.json`, every listed size and SHA-256 checksum, the source identity shared by the artifacts, and all cross-artifact references. Supplying only the two registries bypasses required integrity and provenance checks and is unsupported.

## Troubleshooting

**Classification coverage is incomplete**

Inspect the unresolved indices in the activity log. An incomplete classification is not published, so there is no final bundle audit to inspect. The run fails rather than guessing a style.

**Input exceeds the classifier limit**

The current classifier uses one slim-bundle request and rejects an estimated input above 150,000 tokens. Reduce the template to the actual specification template content before retrying.

**A checksum or bundle validation fails**

Treat the bundle as altered or incomplete. Re-run Phase 1 from the original DOCX and hand Phase 2 the newly published `.phase1` directory intact.

**The source changed while being snapshotted**

Finish saving the document in Word, then retry. Phase 1 will not analyze a mixture of file versions.

**Header/footer media exceeds a limit**

Reduce the embedded asset size in the source template. Limits are deliberate protections against unexpectedly large or hostile package content.
